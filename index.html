Part I: The Design Process

Phase 1: Empathize & Define

This phase is about understanding your users and defining the problem you need to solve. All methods here focus on gathering insights and building empathy.

Glossary for this Phase

  • User Persona: A detailed, semi-fictional profile of a typical user, based on real research. It helps you design for a person, not a statistic.
  • Proto-Persona: A persona created using the existing knowledge of stakeholders, not new research. It's a "best guess" to get started quickly.
  • Empathy Map: A snapshot of a user's world, organized into what they Say, Think, Do, and Feel. It's a tool to build a deeper understanding of the user's mindset.
  • User Journey Map: A timeline of a user's experience (actions, thoughts, feelings). Its main purpose is to find the emotional low points (frustrations) that need fixing.
  • User Story: A simple sentence describing a feature from the user's perspective: "As a [persona], I want to [action], so that I can [benefit]."
  • Pain Point: A specific, frustrating problem a user experiences while trying to use a product. It's the "pebble in their shoe" that your design should try to remove.

1.1 Method: User Personas

Logic: To create a realistic and relatable representation of your key audience segments. This ensures the entire team is focused on the same user.

Actions:

  • Conduct user research (interviews, surveys).
  • Analyze the data to find patterns and group users.
  • Create a detailed profile for each group.

Logic: To quickly align a team on a shared hypothesis about users when there is no time or budget for initial research.

Actions:

  • Gather stakeholders in a workshop.
  • Brainstorm user characteristics based on existing knowledge.
  • Sketch out a basic persona.
  • Acknowledge that it must be validated later.

1.5 Method: User Stories

Logic: To translate user needs into actionable development items, ensuring that every feature built is directly tied to user value.

Actions:

  • Identify a need or opportunity from your research.
  • Frame it using the standard template.
  • Use the resulting stories to build your product backlog.
User Story Template & Example

As a [Persona], I want to [Action / Feature], so that I can [Benefit / Goal].

Example: "As Anxious Alex, I want to be guided through registration step-by-step, so that I can feel confident I'm not making any mistakes."

Phase 2: Ideate & Evaluate

This phase is about generating solutions and making decisions. The methods here help you structure your product and decide what to build first.

Glossary for this Phase

  • Information Architecture (IA): The blueprint for your app's structure. It's how you organize everything so it's not a confusing mess.
  • Card Sorting: A research method where you ask users to group topics to help you design an IA that matches how they think.
  • Tree Testing: A method to validate your IA by asking users to find information in your proposed sitemap.
  • Prioritization Framework: A structured method (like MoSCoW or a Value vs. Effort Matrix) used to help a team decide what to work on first.
  • DFV Framework: A method for evaluating ideas based on three criteria: **D**esirability (do users want it?), **F**easibility (can we build it?), and **V**iability (should we build it?).

2.1 Method: Information Architecture

Logic: To create a logical and intuitive structure for your app, making it easy for users to find what they need.

Actions:

  • List all content and features required for the app.
  • Group related items together into logical categories.
  • Visualize this structure as a sitemap.
Sitemap Template & Example

2.2 Method: Card Sorting

Logic: To design an IA that matches your users' mental models, not just your own assumptions.

Actions:

  • Create a "card" for each piece of content.
  • Ask users to group the cards in a way that makes sense to them.
  • Analyze the groupings to inform your sitemap.
Card Sorting Types
  • Open: Users create their own category labels. (Use to discover a structure).
  • Closed: Users sort cards into categories you provide. (Use to validate a structure).
  • Hybrid: A mix of both, allowing users to create new categories if needed.

2.3 Method: Tree Testing

Logic: To directly test the findability of your IA before you build any UI. It answers the question: "Can users find things in the structure I've designed?"

Actions:

  • Provide users with your sitemap (as a text-based tree).
  • Give them a task (e.g., "Find the registration deadline").
  • Track their clicks to see if they navigate to the correct location.
Example Tree Test Task

"You need to find out the last day to drop a class without a penalty. Starting from the homepage, where would you click?"

2.4 Method: MoSCoW Prioritization

Logic: To force tough decisions about what is truly essential for a product release, ensuring the team delivers a functional and valuable product quickly.

Actions:

  • List all potential features.
  • Categorize each feature as a Must-Have, Should-Have, Could-Have, or Won't-Have.
  • Focus development efforts on the Must-Haves first.
MoSCoW Example
Must-Have: Login, Course Catalog
Should-Have: On-Screen Glossary
Could-Have: "Rate My Professor"
Won't-Have: Social Features

2.5 Method: Value vs. Effort Matrix

Logic: To prioritize features by balancing the value they provide to users against the effort required to build them.

Actions:

  • For each feature, estimate its value (High, Medium, Low) and the effort to implement it (High, Medium, Low).
  • Plot the features on a 2x2 matrix.
  • Prioritize the features in the "High Value, Low Effort" quadrant first.

High Value, High Effort(Major Projects)

Low Value, High Effort(Avoid)

2.6 Method: DFV Framework

Logic: To ensure that a feature idea is well-rounded and sustainable by evaluating it from three critical perspectives: the user, the technology, and the business.

Actions:

  • Assess the feature's **Desirability** by reviewing your user research.
  • Assess its **Feasibility** by consulting with the development team.
  • Assess its **Viability** by analyzing the business case with stakeholders.

Phase 3: Design & Prototype

This phase is about turning your ideas into tangible designs. The methods here help you create the blueprint and interactive models of your app.

Glossary for this Phase

  • Wireframe: A low-fidelity, basic blueprint of a screen's layout. It focuses on structure and functionality, ignoring visual design like colors and fonts.
  • Prototype: An interactive, clickable model of your app. It's used to test the flow and usability of a design before it's built.
  • Fidelity: The level of detail and realism of a prototype. Low-fidelity (lo-fi) is basic and sketchy (like wireframes), while high-fidelity (hi-fi) looks and feels like the final product.
  • Heuristic Evaluation: An inspection method where a design is checked against a list of established usability principles (Heuristics) to find common problems.
  • Usability Testing: The practice of observing real users as they try to complete tasks using your prototype to see where they struggle and get confused.
  • Iteration: The cyclical process of refining a design based on feedback. You test, make changes, and test again until the design is solid.
  • A/B Testing: A method of comparing two versions of a design (A and B) against each other with real users to see which one performs better against a specific metric.

3.1 Method: Wireframing

Logic: To define the structure, content placement, and functionality of screens before investing time in detailed visual design.

Actions:

  • Start with quick, rough sketches of key screens on paper.
  • Create more detailed, low-fidelity digital wireframes using simple shapes and placeholder text.
  • Link the screens together to create a simple, clickable prototype for feedback.

3.2 Method: Heuristic Evaluation

Logic: To find and fix common usability problems in your design early, using a set of established principles as a checklist. It's a fast and cheap way to improve your design before user testing.

Actions:

  • Familiarize yourself with Nielsen's 10 Usability Heuristics.
  • Go through your wireframes screen by screen, checking for violations.
  • Log each issue found, noting the heuristic violated and its severity.
  • Consolidate and prioritize the findings to create an action plan for fixes.

3.3 Method: Prototyping

Logic: To create an interactive, clickable model of your app that allows you to test the flow and usability of the design before it is coded. This saves significant time and resources by catching issues early.

Actions:

  • Choose your fidelity (low or high). For testing flow, start with low-fidelity.
  • Import your static wireframes or mockups into a prototyping tool (e.g., Figma, Adobe XD).
  • Create "hotspots" to link buttons and interactive elements to their corresponding screens.

3.4 Method: Usability Testing

Logic: To get a reality check on your design by observing real users as they interact with your prototype. This is the best way to find out what's confusing or difficult before you build the final product.

Actions:

  • Create a test plan outlining your objectives, participants, and tasks.
  • Conduct the test session: facilitate, observe, and take detailed notes.
  • Analyze the findings to identify patterns and prioritize issues.
  • Create a report with actionable recommendations for design improvements.
Standard Usability Testing Methods
Method 1: Moderated Testing

A test session where a facilitator is present with the participant, either in-person or remotely, to guide them and ask follow-up questions.

Use Cases:

  • Complex Workflows: When the task is long or has many steps, a moderator can keep the user on track.
  • Early Prototypes: To understand the "why" behind user confusion with a new or unfinished concept.
Example & Script Template

Example: In our wizard test, a moderator could ask Alex, "I noticed you paused before confirming. Can you tell me what you were thinking at that moment?"

Script Snippet: "I see you're looking at the filter options. What are your expectations for what will happen when you click that?"

Method 2: Unmoderated Testing

A test where the participant completes tasks alone using an online tool that records their screen and voice. Instructions are provided by the tool, not a person.

Use Cases:

  • Simple Tasks: To validate if users can complete a straightforward task, like a login process.
  • Quantitative Data: To gather metrics like success rate or time-on-task from a larger group of users.
Example & Task Template

Example: We could ask 50 students to use our prototype in an unmoderated test to measure what percentage can successfully register for a class in under 3 minutes.

Task Prompt Template: "Your goal is to register for the course 'Computer Science 101'. Please think out loud as you complete this task."

3.5 Method: Iterating and Refining Your Design

Logic: To create a cycle of continuous improvement by refining your design based on feedback. This process of failing early and cheaply with a prototype ensures you succeed with the final, expensive-to-build product.

Actions:

  • Prioritize the findings from your usability test, focusing on the most severe issues.
  • Brainstorm and decide on the best solutions for the top-priority problems.
  • Implement the changes in your prototype.
  • Conduct another, smaller round of testing to validate that your fixes were successful.

3.6 Method: A/B Testing

Logic: To make objective, data-driven decisions between two design variations by measuring which one performs better at achieving a specific, measurable goal (like getting more clicks).

Actions:

  • Define a clear goal and metric (e.g., increase sign-ups).
  • Create two versions of your design (A and B) that differ in only one way (e.g., button color).
  • Randomly show version A to 50% of your live users and version B to the other 50%.
  • Analyze the data to see which version led to more conversions and declare a winner.

Phase 4: Handoff & Implementation

This final phase is about preparing your tested designs for development and ensuring a smooth transition to the engineering team.

Glossary for this Phase

  • Design Handoff: The process of delivering your final design documentation to the development team to ensure the product is built exactly as designed.
  • UI Kit / Style Guide: A central document that defines all reusable components (buttons, colors, fonts) to ensure design consistency.
  • Design Specifications (Specs): The detailed measurements and properties of every UI element, such as size, spacing, color codes, and font styles.
  • Annotated Mockup: A final design screen with added notes that explain interactions, specifications, or other rules for the developer.

4.1 Method: Design Handoff

Logic: To provide developers with everything they need to build the product to your exact specifications, preventing guesswork and ensuring a high-quality final product.

Actions:

  • Prepare your design specifications, including a UI Kit and annotated mockups.
  • Create a central "source of truth" where all final assets and documentation are organized.
  • Hold a handoff meeting to walk developers through the designs, explain the rationale, and answer questions.
Design Handoff Checklist
Assets & Specs
  • Final High-Fidelity Mockups
  • Clickable Prototype Link
  • UI Kit / Style Guide
  • Exported Icons & Images
Communication
  • Handoff Meeting Scheduled
  • Link to "Source of Truth" shared
  • Follow-up communication plan established
  • Key user flows and context provided

Part II: Design Practices & Reference

Process Frameworks

Higher-level frameworks that structure the entire design process, often combining many of the individual methods.

Glossary

  • Design Sprint: An intense, five-day process for rapidly solving big challenges, creating new products, or improving existing ones.

5.1 Method: Design Sprints

The 5-Day Design Sprint Plan
Day 1: Map

Understand the problem.

Day 2: Sketch

Individually sketch solutions.

Day 3: Decide

Decide which to prototype.

Day 4: Prototype

Build a realistic prototype.

Day 5: Test

Test with 5 real customers.

Design Systems

Established guidelines and component libraries that ensure products are consistent, high-quality, and feel native to their platform.

Glossary

  • Design System: The complete set of standards, reusable components, and guidelines for a product. It's the ultimate instruction manual for design and development.
  • Metaphor: A foundational idea that informs the design system (e.g., Material Design's use of physical paper and ink).

Core Philosophy: The UI is treated like tangible layers of digital paper and ink, using realistic shadows and motion to create meaning and hierarchy.

Part III: A Guide to Figma

This section documents the successful steps for getting started with UX design in Figma, from understanding the interface to building advanced interactive prototypes.

Module 1: Getting Started

1.1: Core UX Principles
Logic: To design effectively, you must understand the goals of User Experience (UX). It's not just about looks, but about creating products that are functional and enjoyable.
  • Usability: Is the design easy to use and understand?
  • Accessibility: Can people with different abilities use it?
  • Desirability: Is the design pleasant and enjoyable to use?
1.2: The Figma Workspace
Logic: Knowing your workspace is the first step to using any tool effectively. Figma has four main areas.

Steps:

  1. Identify the Canvas (center): Your main design area.
  2. Identify the Layers Panel (left): Organizes every object in your design.
  3. Identify the Properties Panel (right): Controls the appearance of any selected object.
  4. Identify the Toolbar (top/bottom): Contains your primary tools (shapes, text, etc.).
Sub-Step: Understanding Pages
Logic: For large projects, a single canvas can become messy. Pages allow you to organize your work into different sections within the same file (e.g., one page for wireframes, another for final designs).

Action: Pages can be created and managed within the Layers Panel.

Sub-Step: Troubleshooting the Interface
Logic: If the interface doesn't look as expected, it's crucial to identify the environment you're in. A bottom toolbar on a computer is the key indicator of a FigJam file.

Action: Check the browser's address bar. figma.com/file/... is a Figma Design File (Correct for our lessons). figma.com/jam/... is a FigJam File (A digital whiteboard with a different interface).

1.3: Canvas Navigation
Logic: You must be able to move around your design quickly and efficiently.

Steps:

  1. Zoom: Hold Ctrl (Windows) or Cmd (Mac) and scroll your mouse wheel.
  2. Pan (Move): Hold the Spacebar (your cursor becomes a hand) and drag your mouse.

Module 2: Your First UI Element

2.1: Understanding Frames
Logic: You must always design within a defined screen size. A Frame acts as the container for each screen of your design (e.g., an iPhone screen).

Steps:

  1. Select the Frame tool (# icon) from the toolbar.
  2. In the Properties Panel on the right, choose a preset size (e.g., iPhone 14).
2.2: Creating & Styling Shapes
Logic: The appearance of every object is controlled from the Properties Panel.

Steps:

  1. Select the Rectangle tool from the toolbar and draw a shape on your canvas.
  2. To Change Color: With the shape selected, find the Fill section in the Properties Panel, click the color swatch, and choose a new color.
  3. To Add a Border: Find the Stroke section and click the + icon. You can then change its color and thickness.
  4. To Add a Shadow: Find the Effects section and click the + icon. The default is "Drop shadow," which adds depth.
2.3: Adding & Styling Text
Logic: Text is a fundamental part of any design. It is created with the Text tool and styled similarly to shapes.

Steps:

  1. Select the Text tool (T icon).
  2. Click on the canvas and type your text.
  3. With the text box selected, use the Text section in the Properties Panel to change properties like font size and weight (e.g., "Bold").
Sub-Step: Troubleshooting Style Changes
Logic: If style changes aren't applying, it's because Figma is in "typing mode" instead of "object selection mode." You must select the entire object to style it.

Action: Click your text, then press the Esc (Escape) key. A blue box will appear around the text, confirming the object is selected. Now you can apply styles from the Properties Panel.

2.4: Aligning Elements
Logic: For a professional look, elements must be perfectly centered, not aligned by eye.

Steps:

  1. Select the first object (e.g., your text).
  2. Hold the Shift key and select the second object (e.g., your rectangle).
  3. In the Properties Panel, a new section of alignment tools will appear at the top. Click "Align horizontal centers" and "Align vertical centers".

Module 3: Building Smart Components with Auto Layout

3.1: The Correct Way to Build an Auto Layout Button
Logic: Auto Layout creates smart frames that automatically resize with their content. The most robust method is to build the component from the text outwards, making the background a property of the frame itself, not a separate object.

Steps:

  1. Start with only a text layer.
  2. Select the text layer and press Shift + A. This creates a new Auto Layout frame perfectly wrapped around the text.
  3. With the new frame selected, go to the Properties Panel and in the Fill section, click + to add a background color.
  4. Test it by double-clicking the text and changing its length. The frame will resize automatically.
3.2: Controlling Space: Padding
Logic: Padding creates "breathing room" between the content and the edge of its frame.

Steps:

  1. Select the Auto Layout frame.
  2. In the Properties Panel, find the Auto Layout section.
  3. Adjust the padding around items. This is typically split into two boxes: Horizontal Padding and Vertical Padding.
3.3: Controlling Space: Gap
Logic: The "Gap" (or "spacing between items") controls the distance between multiple items inside an Auto Layout frame.

Practice Steps:

  1. Create a new shape (e.g., a small circle using the Ellipse tool) to serve as an icon. Remember to hold Shift while drawing for a perfect circle.
  2. Drag this new shape into your Auto Layout button frame, next to the text. The frame will automatically resize to fit it.
  3. Select the main Auto Layout frame.
  4. In the Auto Layout section of the Properties Panel, change the value for "spacing between items" (the gap).

Module 4: Reusable Design with Components

4.1: Creating a Main Component
Logic: To avoid repetitive work and ensure design consistency, you create a single "master" element called a Main Component. Any changes made to this master will automatically apply to all its copies.

Steps:

  1. Select the finished element you want to reuse (e.g., your complete Auto Layout button).
  2. Click the "Create Component" icon (four diamonds in a square) in the top toolbar.
  3. Confirmation: The selected element's outline and its layer name in the Layers Panel will turn purple. This is now your Main Component.
4.2: Creating Instances
Logic: "Instances" are the linked copies of your Main Component that you use throughout your design.

Steps:

  1. Select the Main Component.
  2. Copy it (Ctrl + C or Cmd + C).
  3. Paste it (Ctrl + V) as many times as you need. These new copies are Instances.

Identifying Layers: Main Component: Has a filled, four-diamond icon. Instance: Has a hollow, single-diamond icon [ ‚óá ].

4.3: The Power of Centralized Editing
Logic: The primary benefit of components is that editing the master updates all linked copies instantly.

Steps:

  1. Select the Main Component (the purple one).
  2. Make any change to it (e.g., change the color).
  3. Observe that the change is immediately reflected in all of its Instances.
4.4: Overriding Instance Properties
Logic: Often, you need a variation of a component. Instead of creating a new component, you can "override" the properties of a specific instance without breaking its link to the master.

Steps:

  1. Select any Instance.
  2. Select an element inside that instance (e.g., the text layer).
  3. Change a property (e.g., edit the text content).
  4. Observation: Only the specific instance you edited will change.
Sub-Step: Renaming Layers for Clarity
Logic: When you override text content in an instance, Figma doesn't update the layer's name. It is good practice to rename it manually to keep your file organized.

Action: In the Layers Panel, double-click directly on the layer's name, type the new name, and press Enter.

Module 5: Interactive Prototyping

5.1: Preparing Your Screens
Logic: To create an interactive flow, you need a starting point and a destination. This means you must have at least two separate Frames (screens).

Steps:

  1. Select the entire screen Frame by clicking its name in the Layers Panel.
  2. Press Ctrl + D (or Cmd + D) to duplicate the Frame.
  3. Make a clear visual change to the second Frame (e.g., change its Fill color in the Properties Panel).
5.2: Creating a Prototype Connection
Logic: After preparing your screens, you need to switch to Prototype mode to tell Figma how to connect them.

Steps:

  1. In the Properties Panel, click the "Prototype" tab.
  2. On your first Frame, click the trigger element (e.g., your button).
  3. A small white circle will appear. Click and hold this circle.
  4. Drag the arrow that appears over to your second Frame and release.
5.3: Presenting and Testing
Logic: "Present" mode allows you to interact with your prototype from an end-user's perspective.

Steps:

  1. In the top-right corner, click the "Present" button (a ‚ñ∂Ô∏è play icon).
  2. A new browser tab will open. Click on your trigger element.
  3. Confirmation: The screen should transition to your second Frame.

Module 6: Advanced Component Techniques

6.1: Creating Component Variants
Logic: Variants allow you to group and organize different states of a component into a single container called a "Component Set." This is essential for managing complex design systems.

Steps:

  1. Select your Main Component. In the Properties Panel, find Properties and click +, then choose "Variant".
  2. Rename the default "Property 1" to something descriptive, like "State".
  3. Select the variant inside the new dashed border and click the purple + icon to add a new variant.
  4. Style the new variant (e.g., a lighter color for a hover state).
  5. In the Properties Panel, rename its "State" value to something descriptive, like "Hover".
  6. Drag an Instance from the Assets Panel. Use the "State" dropdown in the Properties Panel to switch between your variants.
6.2: Creating Interactive Components
Logic: This feature allows you to build prototype interactions directly into the Main Component. As a result, all instances will automatically have that interactivity.

Steps:

  1. Work inside your purple dashed Component Set.
  2. Switch to the "Prototype" tab.
  3. Drag a connection handle from the "Default" variant to the "Hover" state variant.
  4. In the "Interaction details," change the trigger from "On click" to "While hovering".
  5. Test in Present Mode. Hovering over any instance will now trigger the state change.

Module 7: Advanced Prototyping and Animation

7.1: Smart Animate for Fluid Transitions
Logic: Smart Animate creates high-fidelity transitions by finding layers with matching names on two different frames and automatically animating the difference in their properties (position, size, color, etc.).

Steps:

  1. Set up two identical frames with an instance of a component in each.
  2. Change a property of the instance on the second frame (e.g., move it).
  3. In the "Prototype" tab, drag a connection from the first frame to the second.
  4. In the "Interaction details," change the animation from "Instant" to "Smart animate".
7.2: Prototyping with Overlays
Logic: Overlays are used for elements that appear on top of an existing screen, such as pop-up modals or menus.

Steps:

  1. Design your pop-up as a separate Frame on the canvas.
  2. Drag a prototype connection from a trigger element on your main screen to the overlay frame.
  3. In the "Interaction details," change the action from "Navigate To" to "Open overlay".
  4. Configure the position and check "Add background behind overlay" to dim the screen.

Module 8: Creating Scrolling Content

8.1: Building the Scrolling Row
Logic: Create a long, horizontal row of components that is wider than the screen to create the need for scrolling.

Steps:

  1. On your main design screen, drag multiple instances of a component (e.g., a card).
  2. Select all instances and apply Auto Layout (Shift + A).
  3. Set the direction to Horizontal and adjust the gap.
8.2: Creating the Viewport and Enabling Scrolling
Logic: The key to scrolling is to place your long content inside a smaller frame (the "viewport") and then enable scrolling on that viewport.

Steps:

  1. Use the Frame tool (#) to draw a viewport frame that defines the visible area.
  2. In the Layers Panel, drag the long Auto Layout row of cards inside this new viewport frame.
  3. Select the viewport frame. In the Properties Panel, ensure "Clip content" is ticked.
  4. Switch to the "Prototype" tab, and with the viewport frame selected, set "Overflow" to "Horizontal scrolling".

Module 9: Advanced Scrolling Techniques

9.1: Creating a Vertically Scrolling Page
Logic: To make a page scroll, the main frame's height must be longer than the device's viewport.

Steps:

  1. Create a standard device Frame.
  2. In the Properties Panel, increase its Height (H).
  3. Place content that extends beyond the initial view.
  4. Select the entire long frame, switch to the "Prototype" tab, and set "Overflow" to "Vertical scrolling".
9.2: Creating a Fixed ("Sticky") Element
Logic: To make an element like a header stay in one place while the rest of the page scrolls, change its position property to "Fixed".

Steps:

  1. Design your header and place it at the top of your long frame.
  2. Select the header element.
  3. Switch to the "Prototype" tab.
  4. Find the "Position" section and change the setting to "Fixed (stay in place)".

Module 10: Creating a Style System

10.1: Creating Color and Text Styles
Logic: Saving colors and text properties as "Styles" allows you to reuse them globally, ensuring consistency. If the style is updated, every element linked to it will also update.

Steps:

  1. Select an object with the color/text properties you want to save.
  2. In the Properties Panel (Fill or Text section), click the four-dot Style icon.
  3. Click the plus + icon to "Create a new style".
  4. Give the style a semantic name (e.g., brand/primary or headings/h1) and click "Create style".

Module 11: Creating and Using Color Palettes

11.1: Creating a Palette with a Naming Convention
Logic: Use a forward slash (/) in your style names to automatically group them into folders in the Styles menu, creating an organized palette.

Steps:

  1. Create a new color style.
  2. In the name field, type the folder name, a slash, and then the color name (e.g., genre/action).
  3. Repeat for other colors, keeping the folder name consistent (e.g., genre/comedy).

Module 12: Professional Design Handoff

12.1: Part 1 - Cleaning & Organizing Your File
Logic: Before sharing, the file must be clean, organized, and easy to navigate. This builds confidence, saves the engineer time, and prevents mistakes that can arise from a messy or confusing file.
Step 1.1: Consistent and Logical Naming
Logic: Layer names are critical because they directly inform the engineer what to name the components in the code. A clear naming convention (Primary Button - Login) is infinitely more useful than a default name (Frame 1028).

Action: Go through the Layers Panel and systematically rename all layers, groups, and frames to be descriptive of their content and function.

Step 1.2: Ungroup Unnecessary Groups
Logic: Overly nested layers (a group inside a group inside a frame) make the design structure unnecessarily complex for an engineer to read. A flatter, simpler layer hierarchy is easier to translate into code.

Action: Identify any group or frame that contains only a single item. Select this unnecessary container and ungroup it using the shortcut Ctrl + Shift + G (or Cmd + Shift + G on a Mac) to simplify the structure.

Step 1.3: Organize Pages and Delete Unused Elements
Logic: The file's structure should guide the engineer directly to the final, approved designs, hiding all drafts and explorations to avoid confusion.

Action:

  1. Create a clear page structure. A professional standard includes pages like "‚úÖ Final Designs", "Components", and an "Archived/WIP" page.
  2. Move your final, cleaned-up screens to the "‚úÖ Final Designs" page.
  3. In the Layers Panel of your final screens, find any hidden layers (eye icon with a slash). If they are not being used for a prototype animation, delete them to reduce clutter.
12.2: Part 2 - Annotating & Specifying
Logic: An engineer cannot know the designer's intent for non-visual details like animations or specific behaviors. Annotations provide these explicit instructions directly within the design file.
Step 2.1: Visually Documenting User Flows
Logic: While prototype links show the connection between screens, adding text labels makes the user flow immediately obvious without needing to enter Present mode.

Action: Use the Text tool (T) to place a label next to each prototype arrow ("noodle") that clearly describes the interaction (e.g., "On Tap: Successful Login").

Step 2.2: Marking Assets for Export
Logic: Engineers need clean, high-quality assets (icons, logos, images) to build the UI. You must mark which layers are exportable so they can be easily downloaded.

Action:

  1. Select the layer that needs to be an asset (e.g., an icon).
  2. In the Properties Panel, scroll to the bottom and find the "Export" section.
  3. Click the plus + icon.
  4. Choose the appropriate format (e.g., SVG for icons and logos, PNG or JPG for images).
Step 2.3: Adding Written Specifications
Logic: Details about animations, validation rules, or other specific behaviors must be written down to ensure they are built correctly.

Action: Use the Text tool (T) to add notes on the canvas next to the relevant elements. For example, next to an animation, write "Animation: Smart Animate, Ease Out, 600ms".

12.3: Part 3 - Sharing & Inspecting
Logic: The final step is to share the file with the correct permissions and understand the "Inspect" mode that engineers will use to get technical details from your design.
Step 3.1: Sharing the File with the Right Permissions
Logic: Engineers should have "view" access, not "edit" access. This allows them to see all specifications and export assets without the risk of accidentally changing the designs.

Action:

  1. Click the blue "Share" button in the top-right corner.
  2. Set the sharing permission to "Anyone with the link can view".
  3. Click "Copy link" to get the shareable URL.
Step 3.2: Understanding the "Inspect" Mode
Logic: The "Inspect" mode is the bridge between your visual design and the engineer's code. It allows them to get all the technical details they need (sizes, colors, fonts, spacing) without having to ask.

Action:

  1. In the right-side panel, click the "Inspect" tab.
  2. Click on any element in your design.
  3. Observe the panel. It will display all the relevant properties (dimensions, colors, typography), provide code snippets (CSS, Swift, XML), and show a "Download" button for any assets you marked for export.

Appendix A: Common UI Elements

A visual glossary for the standard components that act as the building blocks of any digital interface.

Glossary

  • UI (User Interface): The visual part of an application that the user interacts with.
  • Component: A reusable, self-contained piece of the UI, like a button or a search bar.
  • State: The visual appearance of a component based on user interaction (e.g., hover, disabled).

A.1 Element: Buttons

What they are: Clickable elements that allow a user to trigger an action, like submitting a form, confirming a choice, or navigating.

Best Practices:

  • The most important action on a page should be the most prominent button (Primary).
  • Button labels should be clear, action-oriented verbs (e.g., "Save," "Submit," "Cancel").
  • Ensure buttons have clear 'hover' and 'disabled' states.
Button Types & States

A.2 Element: Input Fields

What they are: Form elements that allow users to enter information, such as text, numbers, or selections.

Best Practices:

  • Always use a clear label for each input field.
  • Use placeholder text as a hint, not as a label.
  • Use the correct input type for the required data (e.g., 'password' type to mask text).
Input Field Types

What they are: A collection of links and components that help users move between different sections of an application.

A.5 Element: Modals (Pop-ups)

What they are: A dialog box or pop-up window that appears on top of the main page content, requiring the user to interact with it before they can return to the page.

Best Practices:

A.6 Grid & Sizing Conventions

A quick reference guide to the standard conventions for creating clean, consistent, and user-friendly mobile interfaces.

Grid & Layout

8-Point Grid System: All sizes, margins, and padding should be in multiples of 8 (e.g., 8, 16, 24, 32pt).

  • Columns: Mobile designs typically use a 4-column grid.
  • Margins: Space on the left/right edges. Standard is 16pt.
  • Gutters: Space *between* columns. Standard is 16pt.
Sizing & Spacing (Tap Targets)

Minimum Tap Target: Any interactive element must have a minimum tappable area of 44 x 44pt.

  • Buttons (Height): 44pt, 48pt, or 56pt.
  • Icons (Visual Size): 24 x 24pt (but ensure the tappable area is at least 44x44pt).
